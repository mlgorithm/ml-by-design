The previous two chapters established how model claims should be judged and protected. The next modeling question is now operational: once a task and evaluation plan are in place, what is the simplest serious model family that deserves a real test?

Very often, the answer is a linear model. But that answer is only meaningful if the representation makes the relevant structure visible. A linear model can only add weighted pieces of what it is given. If the important structure has not been encoded, the weighted sum cannot rescue the problem. If the representation is strong, however, one weighted sum can go surprisingly far. The model family is therefore not only a choice of formula. It is a test of whether the representation already exposes the signal we care about.

This chapter is therefore not mainly a tour of linear regression and logistic regression. It is the book's representation chapter and its first real baseline chapter. The real question is:

\begin{center}
\emph{Can a good representation plus one weighted sum already solve the task? If yes, why? If not, what missing structure does the failure reveal?}
\end{center}

The chapter's reusable deliverable is a first baseline design sheet. Keep that target in mind while reading: the goal is to make the first serious linear model explicit enough that later model changes become justified rather than impulsive. By the end of the chapter, the reader should be able to say what the model sees, what it predicts, how its coefficients may be read safely, how it will be regularized, and what specific failure would justify leaving linearity.

\section{Opening Dilemma: Is One Weighted Sum Enough?}

Suppose we have already framed an early student-support task, chosen an evaluation plan, and agreed on what counts as a useful prediction. A support team wants to use early quizzes, submissions, and platform activity to estimate which students may need timely human outreach. We still need a first model family. A deep model may eventually win, but it is rarely the right first move. The right first move is usually the simplest serious model that can reveal whether the representation already contains most of the signal.

That is why linear models appear early in this book. They are not here because they are historically old or mathematically easy. They are here because they make the route from representation to prediction unusually visible. A linear model says, in effect: encode the example as numbers, attach a weight to each one, add the contributions, and transform the result if needed. The simplicity is pedagogically valuable because it makes failure diagnostic instead of mysterious.

Keep that same early student-support setting in view through this chapter. Later chapters will widen the institutional workflow into message routing, retrieval, and grounded answers, but the first baseline question is simpler: can a transparent weighted-sum model already surface the right cases for human review? Each major section of the chapter turns that one question into a modeling discipline: choose the representation, understand the score, read the coefficients carefully, control complexity, and diagnose failure structurally.

\begin{evidencebox}
CDC guidance for Youth Risk Behavior Survey trend analysis still uses logistic regression for dichotomous risk behaviors and linear regression for continuous ones \citep{cdc_yrbs_trends}. These models are not just teaching artifacts. They remain standard analytical tools in serious public-health workflows.
\end{evidencebox}

\section{Representation Is the Model}

By now the basic linear promise should be clear: the model can only add weighted pieces of what it sees. That means the crucial question comes one step earlier. What exactly does it see? The heart of this chapter is not a fitting rule. It is a representation question.

\begin{definition}[Feature]
A feature is a measurable attribute used to represent an example for a model. Features may be numeric, binary, categorical after encoding, counts, ratios, embeddings, or other derived quantities.
\end{definition}

This definition sounds technical, but its consequences are conceptual. A feature is not just a column in a table. It is a modeling decision. When we choose floor area, age, and distance to transit as features for house-price prediction, we are asserting that these properties may help explain price. When we encode words or character patterns for spam detection, we are asserting that certain patterns in language or formatting may carry useful signal. Feature design is therefore not bookkeeping after the interesting part. It is where a large share of the interesting part already happens.

This is why representation is already part of the model. The model does not see ``a house in a good neighborhood'' or ``a student who is disengaging.'' It sees the encoded representation. Someone must decide which numbers or indicators could plausibly carry the relevant information.

\begin{thinkingtool}
\textbf{Representation audit.} Before fitting a model, ask:
\begin{itemize}[leftmargin=*]
\item What is the raw input?
\item What is the encoded representation?
\item What information is lost by the encoding?
\item What assumptions are being inserted artificially?
\end{itemize}
\end{thinkingtool}

\subsection{Raw Inputs and Feature Maps}

Sometimes the raw input is already numeric and usable. Age, temperature, income, and elapsed time fit that description. In many problems, however, raw inputs must be transformed before a linear model can use them effectively. Text must be turned into counts, indicator variables, or embeddings. Categories must be encoded. Dates may need to become cyclic or calendar-aware features. Interactions may need to be added explicitly if they matter.

Mathematically, this is often written as a feature map $\phi(x)$, which transforms a raw example $x$ into a feature vector:
\[
\phi(x) = [\phi_1(x), \phi_2(x), \ldots, \phi_d(x)]^\top.
\]
The model then operates on $\phi(x)$ rather than directly on the raw input. This notation is useful because it separates two sources of modeling power:
\begin{itemize}[leftmargin=*]
\item the \emph{representation} chosen by the feature map
\item the \emph{fitted weights} learned by the model
\end{itemize}

If the representation is weak, fitting better weights may not be enough. If the representation is strong, even a simple weighted sum can work well.

\begin{practicalnote}
The linear-model workflow is compact: encode the raw input, fit a baseline, inspect the residuals or errors, then ask whether the representation needs interaction terms, scaling, or a different feature map before changing model family.
\end{practicalnote}

\begin{prereqbridge}
If the notation $\phi(x)$ is new, read it as ``the version of the raw input that the model actually uses.'' In many practical problems, this is where most of the modeling judgment lives.
\end{prereqbridge}

From this point forward, the notation will be slightly compressed: unless a distinction matters, \(x\) will usually mean the \emph{encoded feature vector} rather than the raw object. When the distinction matters, it is safer to think in two layers:
\[
x_{\text{raw}} \longrightarrow \phi(x_{\text{raw}}) \longrightarrow w^\top \phi(x_{\text{raw}}) + b.
\]
That small discipline prevents one of the most common beginner confusions in linear modeling. The coefficients do not multiply ``the email'' or ``the apartment'' in the abstract. They multiply the engineered features that someone chose to expose.

\begin{example}[From Raw Input to Feature Vector]
Suppose the raw email is ``FREE cash offer today,'' sent from an unfamiliar domain and containing two links. A simple feature map might turn that raw message into
\[
\phi(x_{\text{raw}}) =
\begin{bmatrix}
1 \\
2 \\
0.8 \\
0
\end{bmatrix},
\]
where the four entries mean urgent-language indicator, number of links, sender-risk score, and whether the sender matches a trusted course domain. The linear model never sees the sentence directly. It sees only these four numbers. If the feature map throws away evidence that later turns out to matter, the weighted sum cannot reconstruct it after the fact.
\end{example}

\subsection{Interaction Features and Basis Expansion}

A linear model applied to raw features can only express additive effects: the contribution of each feature is independent of every other. But many real problems have structure that is inherently interactive. The effect of floor area on apartment price may depend on the neighborhood. The effect of missing assignments on student risk may depend on whether the student is in the first or last week of term.

\emph{Interaction features} make these joint effects explicit. An interaction term for two features $x_1$ and $x_2$ is simply their product $x_1 x_2$. When this product is added as a new coordinate, the linear model can now assign a separate coefficient to the joint effect.

\emph{Basis expansion} generalizes the idea. Any function of the raw inputs can be added as a new feature: squares, square roots, logarithms, indicator functions for threshold conditions, products of three or more features, or Fourier-style sinusoidal terms. The resulting feature map $\phi(x)$ may be much richer than the original input, but the model on top of it remains linear in the expanded features.

\begin{example}[Interaction Features in Practice]
Suppose we predict student risk from two features: fraction of assignments missed ($x_1$) and week of semester ($x_2$). Independently, missing a moderate fraction of assignments may be less alarming in week two than in week ten. Adding the product $x_1 x_2$ gives the model a way to express that the effect of missed assignments grows with time. The coefficient on $x_1 x_2$ will be positive if late-semester missing work is especially predictive, regardless of what the individual coefficients on $x_1$ and $x_2$ say.
\end{example}

\begin{mathlens}
If the feature map is $\phi(x) = [x_1,\, x_2,\, x_1^2,\, x_2^2,\, x_1 x_2]^\top$, then the model $w^\top \phi(x) + b$ is nonlinear in the raw input $x$ but still linear in the expanded coordinates $\phi(x)$. Training with gradient descent or ordinary least squares still applies without modification. The representational power comes from the engineer's choice of $\phi$, not from a more complex model family.
\end{mathlens}

Why does this matter? Many students conclude too early that linear models are only useful for obviously linear worlds. Basis expansion shows that a thoughtful feature map can extend a linear model's expressiveness considerably. The cost is interpretability: once features interact, individual coefficients become harder to read as standalone effects. The judgment call is whether the gain in fit justifies the loss in narrative clarity.

\begin{practicalnote}
Add interaction features deliberately rather than exhaustively. Generating all pairwise products of $d$ features produces $d(d-1)/2$ new coordinates. On a dataset with hundreds of features, that explosion can create severe overfitting and slow training. Add interactions when domain knowledge suggests a specific joint effect, and test whether each addition improves held-out performance.
\end{practicalnote}

\begin{example}[A Small Feature Rescue Before Abandoning Linearity]
Suppose an early-support baseline predicts whether a student will need outreach from two raw features: fraction of assignments missed and week of semester. A model that sees only those two raw inputs must pay one global price for missing work. But the same missed-work fraction often means something different in week two than in week ten. If the baseline keeps overpredicting risk early in the term while underpredicting it late in the term, that error pattern is evidence about representation before it is evidence against linearity itself. Adding the interaction \(x_{\text{missed}} x_{\text{week}}\) lets the weighted sum say that missing work matters more as the term advances. A small engineered expansion can therefore rescue the baseline without changing model families.
\end{example}

\subsection{Common Feature Types}

Numeric features are the simplest case. If we predict house prices, a feature like floor area may enter directly as a number. But even then, choices remain. Should we use total area or the logarithm of area? Should we center and scale it? Should we include an interaction between area and neighborhood?

Categorical features require more care. A feature like neighborhood, major, product category, or browser type cannot be inserted directly as a single integer without creating an artificial ordering. A common solution is one-hot encoding, in which each category receives its own indicator variable. In models with an explicit intercept, practitioners usually drop one indicator as the reference category or use regularization carefully so the representation is not perfectly collinear.

Text features are often sparse. A bag-of-words representation may include one feature per word, with each feature recording whether the word appears or how often it appears. This is one reason linear models remain strong in text classification: even though the representation is high-dimensional, the modeling assumption is still a weighted combination of informative signals.

\begin{example}[Feature Design for Housing]
Suppose we want to predict apartment prices. Raw inputs might include address, floor area, year built, floor number, and text descriptions. A reasonable first feature set could contain area, number of bedrooms, age of the building, distance to the city center, and one-hot indicators for neighborhood. This feature set is already making several claims: location matters, age matters, and neighborhood effects should be modeled separately rather than forced into a single numeric code.
\end{example}

\begin{example}[Feature Design Across Public Tasks]
The same logic appears across public tasks. A sports-shot prediction problem suggests structured tabular features such as player position, shot distance, and remaining time. A news topic classification problem suggests sparse lexical features or TF--IDF vectors. The tasks are different, but the chapter's question is the same: what representation would let a weighted sum detect the relevant signal?
\end{example}

\begin{codeexample}[caption={A generic feature pipeline for a linear model}]
raw_example = collect_input()
features = feature_map(raw_example)
score = dot(weights, features) + bias
prediction = output_transform(score)
\end{codeexample}

This small program is worth remembering. Linear regression and logistic regression are two specific instances of this general pattern. The main difference is what the output transform does. The score-producing core stays the same, which is why the chapter can teach both models as one reusable pipeline rather than as two unrelated formulas.

\begin{thinkingtool}
\textbf{Structural rescue test.} If a linear baseline fails, ask two questions in order:
\begin{itemize}[leftmargin=*]
\item Could one additional transformed feature make the missing pattern visible, such as area-squared or attendance times homework score?
\item If not, what kind of structure is missing entirely: local neighborhoods, branching rules, sequence order, or a learned representation?
\end{itemize}
Do not jump too quickly from ``the line failed'' to ``the whole model family failed.'' First test whether the representation, not the weighted sum, is the real bottleneck.
\end{thinkingtool}

\begin{decisionbox}
\textbf{Which linear family first?}
\begin{itemize}[leftmargin=*]
\item Numeric target such as price, time, or demand: start with linear regression.
\item Binary target or event probability such as spam versus not spam: start with logistic regression.
\item Need multiclass probabilities: the same score-then-transform logic extends, but the chapter's conceptual split remains numeric output versus probability output.
\item Need an action rather than only a score: keep the fitted score, the probability, and the thresholding rule conceptually separate.
\end{itemize}
\end{decisionbox}

\section{Linear Regression as the First Concrete Case}

The generic pipeline becomes easier to trust once it is attached to one concrete model. Linear regression is the simplest place to start because the target is numeric and the weighted-sum structure stays visible all the way through.

\begin{definition}[Linear Regression]
Linear regression predicts a numeric target as an affine function of the features:
\[
\hat{y} = w^\top x + b,
\]
where $x$ is the feature vector, $w$ is the vector of coefficients, and $b$ is the intercept.
\end{definition}

Here again, \(x\) means the representation after encoding. If the notation begins to feel too compressed, mentally expand the same model back to \(\hat{y} = w^\top \phi(x_{\text{raw}}) + b\). That is the cleaner way to remember that linear modeling and feature design are one pipeline, not two disconnected steps.

The word \emph{linear} here refers to linearity in the parameters and the feature representation. This is important. A linear model can still express nonlinear behavior in the original raw input if the features themselves encode nonlinear transformations.

\subsection{Residuals and Squared Error}

For a training example $(x_i, y_i)$, the prediction error is often written as a residual:
\[
r_i = y_i - \hat{y}_i.
\]
Linear regression is commonly fit by minimizing the mean squared residual:
\[
\text{MSE} = \frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2.
\]

Chapter 2 already explained why squared error is a useful training loss. Here the important skill is more concrete: move comfortably from feature values to prediction to residual, and understand what the residual says about the baseline. A positive residual means the model predicted too low; a negative residual means it predicted too high.

\subsection{Minimizing the Loss: Gradient Descent}

For ordinary linear regression, one often solves the least-squares problem directly with linear algebra. Gradient descent is still an important optimization method, especially once the model or loss becomes too large or too complex for a simple closed-form solve. Starting from some initial weight vector $w$, we repeatedly move the weights in the direction that decreases the loss the fastest. The gradient $\nabla_w \mathcal{L}$ points uphill, so we subtract a small multiple of it:
\[
w \leftarrow w - \eta \, \nabla_w \mathcal{L}(w),
\]
where $\eta > 0$ is the learning rate, which controls the step size. For MSE, the gradient with respect to weight $w_j$ is
\[
\frac{\partial \,\text{MSE}}{\partial w_j} = \frac{2}{n} \sum_{i=1}^{n} (\hat{y}_i - y_i) \, x_{ij}.
\]
Each component of the gradient tells one weight which direction to move and by how much. One application of this update is one gradient descent step.

\begin{codeexample}[caption={From scratch: one gradient descent step for linear regression}]
# Student support task: features are (attendance_rate, late_submission_rate),
# target is risk_score (higher means more at-risk)
data = [
    ([0.9, 0.1], 0.2),   # high attendance, few lates -> low risk
    ([0.7, 0.3], 0.4),
    ([0.5, 0.5], 0.6),
    ([0.3, 0.7], 0.7),
    ([0.2, 0.8], 0.9),   # low attendance, many lates -> high risk
]

learning_rate = 0.1
weights = [0.0, 0.0]   # one weight per feature; no explicit bias here
n = len(data)

def dot(a, b):
    return sum(wi * xi for wi, xi in zip(a, b))

# --- compute MSE loss before the step ---
loss_before = sum((dot(weights, x) - y) ** 2 for x, y in data) / n

# --- compute gradient ---
gradient = [0.0] * len(weights)
for x, y in data:
    error = dot(weights, x) - y     # (pred - true)
    for j in range(len(weights)):
        gradient[j] += (2 / n) * error * x[j]

# --- update weights ---
weights = [weights[j] - learning_rate * gradient[j]
           for j in range(len(weights))]

# --- compute MSE loss after the step ---
loss_after = sum((dot(weights, x) - y) ** 2 for x, y in data) / n

print(round(loss_before, 4))   # loss before the step
print(round(loss_after, 4))    # loss after one correction
\end{codeexample}

Running this program produces a smaller loss after a single step. The code makes two things visible at once: the loss goes down because the weights moved in the direction that locally reduced the squared errors, and each component of the gradient tells its corresponding weight how far to move. In this tiny no-bias setup, both weights increase on the first step because the model starts by underpredicting every positive target from an all-zero score. That first-step sign should not be overinterpreted as the final semantic effect of each feature; later updates, or a model with an explicit bias term, separate those effects more clearly.

Imagine a small apartment dataset with features for floor area, number of bedrooms, and age of the building.

\begin{warningbox}
An interpretable model is not the same thing as a correct interpretation. Coefficients are only interpretable relative to the features included, their scales, and the correlations among them.
\end{warningbox}

\subsection{Residual Diagnostics}

Residuals are not just bookkeeping after the fit. If residuals curve with the fitted values, the representation is probably missing nonlinear structure. If their spread grows with the prediction, the error variance may not be constant. If one slice produces extreme residuals, that slice may need a separate feature, a missingness indicator, or a different interaction term. In a linear baseline, residual plots are often the fastest way to learn what structure the representation left out.

\begin{example}[Residual Pattern $\rightarrow$ Representation Repair]
Suppose a housing baseline predicts apartment price from floor area alone and the residual plot shows a U-shape: very small and very large apartments are consistently underpredicted, while mid-sized apartments are overpredicted slightly. That pattern is evidence about representation before it is evidence against linear baselines altogether. A sensible first repair is to add a transformed feature such as area-squared and then ask whether the residual curvature shrinks on validation data. If a different residual pattern appears only on examples with missing renovation-year data, the first repair may instead be a train-only imputation rule plus a missingness indicator. The residual is useful because it points to the next feature decision, not just to the fact that the current fit is imperfect.
\end{example}

\section{Logistic Regression as Score-to-Probability Modeling}

Linear regression predicts unrestricted numeric outputs, which makes it unsuitable for binary classification. If the target is whether an email is spam or whether a student needs support, we usually want a probability or a class label, not an arbitrary real number such as $1.7$ or $-0.8$.

Logistic regression keeps the same weighted-sum backbone and changes only the output transform. The model first computes a score
\[
z = w^\top x + b,
\]
and then maps that score to a probability through the sigmoid function:
\begin{align*}
\sigma(z) &= \frac{1}{1 + e^{-z}}, \\
P(y=1 \mid x) &= \sigma(w^\top x + b).
\end{align*}

In predictive language, this is the model's estimated conditional probability, not automatically a calibrated real-world frequency. This is why the generic pipeline matters. Linear regression uses the identity transform. Logistic regression uses the sigmoid transform. But the change is not only cosmetic. Logistic regression is trained against a classification objective such as log loss, and its output is meant to support probability estimation and threshold policy rather than unrestricted numeric prediction. The reader should keep three layers separate all the way through: score, probability, and decision rule.

Logistic regression is linear in the score, the log-odds, and the decision boundary, not in the final probability output itself.

\subsection{From Score to Probability}

Why use the sigmoid? It maps any real-valued score into the interval $(0,1)$, so the output can be interpreted as a probability, and it preserves ordering: larger scores correspond to larger predicted probabilities.

Another useful viewpoint is through the log-odds:
\[
\log \frac{P(y=1 \mid x)}{1 - P(y=1 \mid x)} = w^\top x + b.
\]
This follows directly from the sigmoid. If $p=\sigma(z)=\frac{1}{1+e^{-z}}$, then $1-p=\frac{e^{-z}}{1+e^{-z}}$, so
\[
\frac{p}{1-p}=e^z
\quad\Longrightarrow\quad
\log\frac{p}{1-p}=z=w^\top x+b.
\]
Students do not need to become obsessed with this form, but it clarifies why logistic regression is mathematically clean: the classification problem is still built from a linear scoring rule.

\begin{warningbox}
Probability and odds are not the same thing. A probability of $0.8$ means ``80\% chance.'' Odds of $4{:}1$ mean four positive cases for every one negative case. Logistic regression becomes easier to interpret once that distinction is kept explicit.
\end{warningbox}

\begin{practicalnote}
A one-unit increase in feature $x_j$ changes the log-odds by $w_j$ and multiplies the odds by $e^{w_j}$, holding the encoded features fixed. That is often the cleanest coefficient interpretation in logistic regression.
\end{practicalnote}

\begin{theorem}[A $0.5$ logistic threshold is a score threshold]
If logistic regression predicts the positive class whenever $P(y=1 \mid x) \ge 0.5$, then this rule is exactly the same as predicting the positive class whenever
\[
w^\top x + b \ge 0.
\]
The boundary between the two decisions is therefore the set of inputs for which
\[
w^\top x + b = 0.
\]
\end{theorem}

\begin{proof}
Let $z = w^\top x + b$. The sigmoid sends $z=0$ to $\sigma(0)=0.5$, and larger scores produce larger probabilities. Therefore
\[
P(y=1 \mid x) \ge 0.5
\quad \Longleftrightarrow \quad
\sigma(z) \ge 0.5
\quad \Longleftrightarrow \quad
z \ge 0.
\]
Substituting back gives the result.
\end{proof}

\begin{mathlens}
For any probability threshold $\tau \in (0,1)$, the equivalent score cutoff is
\[
w^\top x + b \ge \log\frac{\tau}{1-\tau}.
\]
The theorem's $0.5$ case is the special case where this cutoff is zero.
\end{mathlens}

\subsection{Worked Example: A Spam Probability}

Suppose a mail system wants to estimate the probability that a message is spam. Features might include suspicious-link signal, urgent-language signal, and sender-risk score. A logistic model might learn that stronger values on these features push the score upward.

To see the mechanics directly, suppose the feature vector for one message is
\[
x =
\begin{bmatrix}
0.6 \\
0.8 \\
0.4
\end{bmatrix},
\]
where the three components represent scaled signals for suspicious links, urgent sales language, and sender-risk score. Let the fitted logistic-regression parameters be
\[
w =
\begin{bmatrix}
1.4 \\
1.8 \\
1.1
\end{bmatrix},
\qquad
b=-2.0.
\]
The linear score is therefore
\[
z = w^\top x + b = 1.4(0.6) + 1.8(0.8) + 1.1(0.4) - 2.0 = 0.72.
\]
Applying the sigmoid gives
\[
P(y=1 \mid x) = \sigma(0.72) \approx 0.673.
\]

\begin{codeexample}[caption={Algorithm sketch: logistic score and probability}]
Input: feature vector x, weight vector w, bias b
score = dot(w, x) + b
probability = 1 / (1 + exp(-score))
if probability >= 0.5:
    label = 1
else:
    label = 0
return score, probability, label
\end{codeexample}

This tiny program mirrors the worked example exactly. It is worth typing once. The purpose is not software engineering. The purpose is to make the pipeline concrete enough that a student can move from feature values to score to probability to label without hiding behind a library call.

Here ``all else equal'' means holding the encoded features fixed inside the fitted model, not holding the whole world fixed.

\begin{example}[Why Marginal Effects Help]
CDC teaching material on marginal effects emphasizes a point that matters here: changes in predicted probability are often easier to communicate than raw changes in log-odds \citep{cdc_marginal_effects}. Logistic coefficients are mathematically elegant, but many users understand ``the predicted probability rises by about five points'' more naturally than ``the log-odds increase by 0.2.''
\end{example}

For a single feature $x_j$, the corresponding marginal effect comes from the chain rule:
\[
\frac{dP(y=1\mid x)}{dx_j}
=\frac{d\sigma(z)}{dz}\frac{dz}{dx_j}
=\sigma(z)\bigl(1-\sigma(z)\bigr)w_j
=p(1-p)w_j.
\]
So the same coefficient can have a larger or smaller probability impact depending on where the example sits on the sigmoid curve.

\section{Reading Coefficients Without Fooling Yourself}

One reason students are drawn to linear models is that the coefficients seem readable. That instinct is partly right, but it needs guardrails. Coefficients are not self-explaining truths. They are statements about the fitted relationship \emph{within a particular representation}. The next three subsections give the minimum guardrails: scale, correlation, and omitted variables all change what a coefficient can honestly be said to mean.

\subsection{Scale Matters}

If one feature is measured in square meters and another in kilometers, their coefficients will naturally be on different scales. A larger coefficient does not automatically mean a more important feature. Standardizing features can help when we want to compare relative effect sizes more meaningfully, especially in regularized models.

Even standardized coefficients are not robust feature-importance measures when features are strongly collinear or when regularization is active.

This point is easy to underestimate. A coefficient of $-1.4$ per year of building age and a coefficient of $-0.117$ per month describe the same practical effect under different units. Without understanding the units, the raw numbers invite the wrong comparison.

\subsection{Correlation Matters}

If two features are strongly correlated, such as floor area and number of bedrooms, the model may distribute their effects somewhat arbitrarily. Both features may matter, but the exact coefficient values can become unstable. This does not always ruin prediction, but it weakens naive coefficient stories.

In the extreme case of near-perfect collinearity, coefficients can become non-unique or wildly unstable even when predictions barely change. Ridge-style regularization can stabilize the allocation, but it does not restore a unique interpretation by itself.

In practice, this means two models trained on slightly different samples can make similar predictions while assigning noticeably different weights to correlated inputs. Prediction can remain stable while interpretation becomes fragile.

\paragraph{Resampling reveals instability.} One diagnostic for coefficient fragility is to train the same model on bootstrap resamples of the training set and inspect how much each coefficient varies across those resamples. A coefficient that regularly switches sign, or that varies by more than its mean value, should not be interpreted as evidence of a stable directional effect. It may be absorbing the variance of a correlated neighbor. A stable, interpretable coefficient stays in the same direction and roughly the same magnitude across resamples.

\paragraph{Adding or removing a feature changes the story.} Suppose a wage model is trained first with both years of experience and job tenure, then with only job tenure removed. If the coefficient on years of experience changes substantially when tenure is dropped, that is evidence of collinearity. The original coefficient was not measuring the ``pure effect'' of experience; it was absorbing part of the tenure effect as well. This is why coefficient narratives built on a single model fit can be misleading even when the model predicts well.

\begin{warningbox}
Before reporting a coefficient as evidence of a meaningful relationship, check: does the coefficient stay stable when correlated predictors are added or removed? Does it stay stable across resamples of the data? If the answer to either question is no, the narrative is fragile. The model may still predict well, but the specific coefficient value should not carry the story.
\end{warningbox}

\subsection{Missing Variables Matter}

Suppose a wage model includes education but omits years of experience or industry. The education coefficient may absorb some of those missing effects. The model may still have predictive value, but the interpretation of that coefficient becomes less trustworthy.

\begin{example}[A Misleading Coefficient Story]
Imagine a customer-churn model in which the number of refund requests has a positive coefficient. A naive interpretation would be that requesting refunds causes customers to leave. A more plausible explanation is that dissatisfied customers are precisely the ones who request refunds. Predictive association is not causal explanation.
\end{example}

\begin{example}[What Missing Means Can Be Part of the Model]
In an early-support baseline, a missing quiz score can mean at least three different things: the quiz has not happened yet, the student skipped it, or the score failed to enter the system. Treating all three as the number zero collapses different states of the world into one feature value. A better baseline may use an imputed score together with a missingness indicator, or different indicators for ``not yet observed'' and ``missing despite being expected.'' Missingness is therefore part of representation design, not clerical cleanup.
\end{example}

\begin{practicalnote}
If missing values must be imputed, learn the imputation rule on the training split only and then apply that fitted rule unchanged to validation and test. Computing a mean, mode, or fill value on the full dataset quietly leaks evaluation information across the boundary established in Chapter~3. The same discipline applies to standardization, target encoding, and any other preprocessing step that estimates statistics from data.
\end{practicalnote}

\begin{warningbox}
What you are usually allowed to say:
\begin{itemize}[leftmargin=*]
\item ``This feature is associated with higher predicted risk in this model.''
\item ``Holding the encoded features fixed, increasing this variable raises the score.''
\end{itemize}
What you are usually not allowed to say from the model alone:
\begin{itemize}[leftmargin=*]
\item ``This feature causes the outcome.''
\item ``Changing this feature in the world will definitely change the target by this amount.''
\end{itemize}
\end{warningbox}

\begin{practicalnote}
Classical statistical inference asks extra questions that this chapter is not trying to answer in full, such as whether coefficient uncertainty can be estimated cleanly and whether hypothesis tests are justified. For this book, the main use of a linear model is still predictive baseline design, so coefficient-significance language is secondary to prediction quality and interpretation discipline.
\end{practicalnote}

\begin{thinkingtool}
\textbf{Coefficient interpretation checklist.} Before telling a story with a coefficient, pause and ask:
\begin{itemize}[leftmargin=*]
\item What exactly was encoded, scaled, or imputed before this coefficient was fit?
\item Does the coefficient stay qualitatively similar when correlated predictors are added, removed, or resampled?
\item Could missing variables, missingness indicators, or measurement choices be carrying part of the apparent effect?
\item Am I making a predictive association claim or accidentally sliding into a causal one?
\end{itemize}
If those questions do not have stable answers, the model may still be a useful baseline, but the narrative attached to the coefficient should stay modest.
\end{thinkingtool}

\begin{thinkingtool}
\textbf{What can a fitted linear model justify?} Keep the synthesis simple:
\begin{itemize}[leftmargin=*]
\item \emph{Usually safe:} a statement about how the fitted score or predicted risk changes within the encoded representation.
\item \emph{Safe only with caution:} a coefficient story, if it survives resampling, correlated-feature changes, and unit changes.
\item \emph{Not justified yet:} a policy or intervention claim about what would happen if the world were changed.
\end{itemize}
That three-step boundary helps keep prediction, interpretation, and intervention from collapsing into one another.
\end{thinkingtool}

\section{Prediction Is Not Intervention}

Linear models tempt people into causal language because the coefficients look clean. This is the point where linear transparency becomes dangerous if the reader stops being precise. A predictive model describes patterns in observed data: given the features we observed, what outcome should we expect? A causal question is different: what would happen if we deliberately changed one part of the world? Pearl's causal framework is useful mainly because it keeps those two questions separate \citep{pearl2009}.

This distinction matters even in simple tabular problems. A model may learn that students who attend more office hours tend to earn higher grades. That makes office-hour attendance a potentially useful predictive feature. It does \emph{not} by itself prove that forcing attendance to increase would raise grades by the same amount. Students who seek help may also be more motivated, have different schedules, or receive more support elsewhere. Those hidden differences can create confounding.

\subsection{Confounding Changes the Story}

A \emph{confounder} is something that affects both the feature of interest and the outcome. When confounders are missing or measured badly, the fitted coefficient can mix together several reasons for the pattern we see. The model can still predict well, but its coefficients become unsafe to read as advice about what action would improve the outcome.

\begin{example}[Tutoring Is Predictive, but the Policy Claim Is Harder]
Suppose a model finds that students who sign up for tutoring are less likely to fail. That may mean tutoring helps. It may also mean that students who sign up early are the same students who are more organized, more supported, or more engaged in the course. A tutoring coefficient may therefore be useful for prediction while remaining insufficient for the policy claim ``increasing tutoring sign-ups will reduce failure by this amount.'' 
\end{example}

\subsection{What Stronger Causal Language Requires}

To move from a predictive pattern to a causal claim, the evidence standard changes. Stronger causal language usually requires a design that can support claims about what would happen if something were changed: a randomized experiment, a natural experiment with defensible assumptions, or a domain model that states the needed causal assumptions clearly. A fitted linear model on observational data is rarely enough on its own.

The practical rule for this book is conservative. If the task is prediction, say ``linked to,'' ``useful for prediction,'' or ``raises the model score when other encoded features are held fixed.'' Reserve words such as ``causes,'' ``effect,'' or ``would improve outcomes if changed'' for settings where the causal design has actually been defended.

\section{Controlled Simplicity: Regularization}

Once a linear model has enough features, the next risk is overfitting. Even linear models can overfit, especially when the number of features is large or when features are noisy. Regularization addresses this by penalizing overly large or overly complex coefficient vectors. It is the chapter's answer to a practical baseline question: how do we keep the first serious model controlled without making it uselessly weak?

\begin{definition}[L2 Regularization]
L2 regularization adds a penalty proportional to the squared magnitude of the coefficients:
\[
\mathcal{L}(w,b) = \frac{1}{n}\sum_{i=1}^{n}\ell(y_i,\hat{y}_i) + \lambda \|w\|_2^2.
\]
\end{definition}

This penalty discourages extreme weights and tends to spread influence more smoothly across correlated features. In regression, the resulting model is often called \emph{ridge regression}. In classification, the same idea is commonly used with logistic regression.

\begin{definition}[L1 Regularization]
L1 regularization adds a penalty proportional to the sum of absolute coefficient values:
\[
\mathcal{L}(w,b) = \frac{1}{n}\sum_{i=1}^{n}\ell(y_i,\hat{y}_i) + \lambda \|w\|_1.
\]
\end{definition}

L1 regularization often drives some coefficients exactly to zero. This can be useful when we suspect that only a subset of features truly matters, or when we want a sparse model that is easier to inspect.

In most implementations, the intercept $b$ is left unpenalized; the regularizer is meant to control feature weights, not to force the global baseline toward zero.

Conceptually, both penalties express a preference for restrained explanations. They do not ban complexity entirely. They require the data to justify it. That is why regularization belongs to baseline design rather than to after-the-fact cleanup.

The geometry gives a useful picture: the L2 penalty has round contour lines, so optimization usually prefers to shrink many weights smoothly rather than eliminate them. The L1 penalty has diamond-shaped contours with corners on the axes, so solutions are more likely to land exactly on an axis, which is why zeros appear naturally.

\begin{practicalnote}
Regularization strength $\lambda$ is not a matter of aesthetic preference. It is a model-selection choice and should be chosen by validation performance. This is also why standardization matters more once regularization is active: if one feature is measured on a much larger scale than another, the penalty no longer treats them comparably.
\end{practicalnote}

\begin{codeexample}[caption={From scratch: L2 versus L1 gradient penalty}]
# Both penalties are added to the data-loss gradient before each weight update.
# lam is the regularization strength; weights is the current weight vector.

# L2 penalty gradient: proportional to the weight itself (smooth shrinkage)
l2_penalty_gradient = [2 * lam * w for w in weights]

# L1 penalty gradient: proportional to the sign of the weight (sparse shrinkage)
def sign(v):
    return 1 if v > 0 else (-1 if v < 0 else 0)

l1_penalty_gradient = [lam * sign(w) for w in weights]

# Combined update step (choose one penalty):
# weights[j] -= learning_rate * (data_gradient[j] + l2_penalty_gradient[j])
# weights[j] -= learning_rate * (data_gradient[j] + l1_penalty_gradient[j])
\end{codeexample}

The L2 penalty nudges every weight toward zero by an amount proportional to the weight's current value, so large weights shrink faster than small ones but none ever reaches exactly zero. The L1 penalty moves every non-zero weight by the same fixed step regardless of magnitude, which is why it can drive weights all the way to zero and produce sparse models.

\section{Why Linear Models Are the Right First Baseline}

A strong baseline should be cheap to run, understandable to debug, and hard to beat for the right reasons. Linear models satisfy all three criteria. A good baseline is supposed to be beatable eventually, but not trivially. If a richer model wins, it should win by capturing a named piece of structure that the linear baseline could not.

\begin{thinkingtool}
\textbf{Baseline ladder.} Start with the simplest serious model that matches the task. Climb only when you can name the missing structure.
\end{thinkingtool}

On structured tabular data, much of the useful signal may already be present in reasonably chosen features. A linear model can exploit that signal immediately without needing enormous datasets. On sparse text classification, the additive effect of informative words is often enough to perform well. In such cases, if a larger model wins, it should win by a clear and justified margin.

Linear baselines are also diagnostic. If a linear model already performs well, the problem may not require extreme expressiveness. If it performs badly, the failure teaches us something. Perhaps nonlinear interactions matter. Perhaps spatial or sequential structure matters. Perhaps the feature representation is too weak. In each case, the linear model gives us a clean reference point for deciding what kind of additional structure is needed.

\begin{example}[Baseline Logic on Public Tasks]
For a tabular task such as sports-shot prediction, a standardized linear or logistic baseline is often the right first move. For a text task such as news topic classification, bag-of-words or TF--IDF plus logistic regression is still a meaningful reference point. The baseline is useful even when it loses, because the manner of the failure tells us what structure may be missing.
\end{example}

\section{Sparse Linear Models Stay Strong in Text and Tabular Work}

A persistent misconception is that linear models are only a warm-up exercise before the real work with deep models begins. In several important problem families, sparse linear models remain highly competitive not as compromises but as genuine strong baselines.

\textbf{Text classification.} A bag-of-words or TF--IDF representation with logistic regression can achieve classification accuracy on tasks like news topic classification, spam filtering, or sentiment classification that deep models beat by only a few percentage points, at a fraction of the computational cost. The reason is straightforward: in text classification, the vocabulary already exposes the relevant signal. A word like ``inheritance'' is informative about legal topics. A word like ``free'' is informative about spam. A linear model on a well-engineered bag-of-words can identify those associations efficiently.

\textbf{High-dimensional tabular prediction.} In fraud detection, credit scoring, and clinical risk modeling, datasets often have hundreds or thousands of engineered features over millions of examples. Many of those features are individually informative, and their combination is largely additive. A regularized linear model can handle this well. The L1 penalty selects a sparse subset of informative features, making the model both competitive and interpretable. Tree ensembles often win eventually, but rarely by a margin that justifies a tenfold increase in computational cost for the initial prototyping.

\begin{example}[Linear Baseline on a Text Task]
For a news topic classifier over four categories, a logistic regression with TF--IDF features might achieve $93\%$ accuracy. A fine-tuned transformer might reach $96\%$. That $3$-point gap may matter for a production system, but it does not mean the linear baseline was uninformative. It established that the task is solvable, identified which categories confuse the model, and provided a reference point against which the expensive model must justify its cost.
\end{example}

\begin{thinkingtool}
\textbf{Baseline ladder in text and tabular work.} Before training any deep model, ask: does a regularized logistic or linear regression on engineered features already work well? If it does, the deep model must provide either a meaningful accuracy gain on the evaluation set or a well-justified argument for why it will perform better in production. Neither argument is hard to make, but both should be made explicitly.
\end{thinkingtool}

The broader point is that linear models expose the relationship between representation and prediction transparently. When they succeed, they tell you something about the structure of the problem: it was largely additive in the feature space you chose. When they fail by a large margin, they tell you something even more useful: the problem contains interactions, local geometry, or sequential structure that the feature set could not capture. Either result is informative. The deep model that follows should be chosen in response to that specific structural diagnosis.

\section{Failure Diagnosis, Not Method Shopping}

Sooner or later, a linear model will miss something important. When that happens, the useful question is not ``what model is fashionable now?'' It is ``what kind of structure is this failure pointing to?'' No model family is universal. Linear models struggle when the decision boundary or regression surface cannot be captured well by a weighted sum of the available features. Images, audio, and raw language sequences often contain local, hierarchical, or contextual structure that a simple linear score cannot exploit directly unless those structures have already been encoded by other means.

They also struggle when interactions are essential but omitted. If the effect of one feature depends strongly on another, and that interaction is not encoded explicitly, a linear model may be systematically wrong even when each feature is individually informative.

This is the right way to think about moving beyond linear models. Do not abandon them because a richer family is fashionable. Abandon them only when the problem or the evidence shows that the needed structure cannot be captured adequately in the current representation.

\section{Chapter Artifact: A First Baseline Design Sheet}

By the end of this chapter, the reader should be able to write down the first defensible baseline before trying more expressive methods. The point is not to freeze creativity. The point is to make the first modeling move explicit enough that later changes become justified rather than impulsive. The framing memo named the decision context, the metric worksheet named good behavior, and the evaluation plan protected the claim. This sheet names the first serious model that deserves to make that claim. It is where representation audit and baseline ladder turn into one written modeling commitment.

\section{Common Mistakes with Linear Models}

Once the baseline sheet is clear, the most useful closing move is to name the habits that regularly weaken linear-model work. Most of them are not about the formula itself. They are about representation, interpretation, and evaluation discipline.

\begin{mistakebox}
\item \textbf{Treating feature engineering as mechanical.} For linear models, representation is part of the model.
\item \textbf{Overinterpreting coefficients.} Coefficients describe the fitted model, not the world's causal structure.
\item \textbf{Encoding categories as ordered integers.} Use one-hot encoding for nominal categories unless order is real.
\item \textbf{Ignoring missingness as information.} Missing values may be signal, so impute on the training split and add a missingness indicator when needed.
\item \textbf{Ignoring standardization.} Scale affects optimization and coefficient comparison, especially with regularization.
\item \textbf{Skipping the linear baseline.} A simple linear baseline is a useful diagnostic, even when you expect a richer model.
\item \textbf{Forgetting the evaluation boundary.} Preprocessing must respect train, validation, and test splits.
\item \textbf{Assuming linearity means simplicity of the world.} A linear model can fit a complex problem only if the representation makes it approximately linear.
\end{mistakebox}

\section{Looking Ahead}

This chapter showed that representation is already part of the model. The next chapter studies what kind of structure is missing when one global weighted sum is not enough: conditional thresholds and interactions, local similarity, or unlabeled organization.

The first baseline design sheet from this chapter becomes the starting document for Chapter 5's structure diagnosis sheet: if the linear baseline fails, the written failure pattern should justify the next family rather than model shopping.

\begin{chaptersummary}
\item This chapter begins with a representation question: can a good encoding plus one weighted sum already solve the task?
\item The model does not see raw reality. It sees an encoded representation, so feature choices are already scientific claims about what signal matters.
\item Linear regression and logistic regression share one pipeline: encode the input, compute one weighted sum, then choose the output transform that matches the target.
\item Coefficients become useful only under discipline about scale, correlation, omitted variables, and the boundary between predictive language and causal claims.
\item Regularization keeps the first baseline controlled: L2 shrinks weights smoothly, while L1 can also make the model sparse.
\item Linear models are strong first baselines because they are cheap, legible, and diagnostically useful.
\item Failure should be interpreted structurally: first ask whether a better feature map rescues the weighted sum, and only then ask what richer family is justified.
\item The main deliverable of the chapter is a first baseline design sheet written before method shopping begins.
\end{chaptersummary}
