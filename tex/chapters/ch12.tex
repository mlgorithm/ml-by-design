\item Understand why a deployed AI system must be designed as a pipeline with requirements, interfaces, monitoring, and fallback behavior rather than as a model checkpoint alone.
\item Distinguish offline model quality from system quality, including latency, capacity, coverage, calibration, and review workflow constraints.
\item Explain training-serving skew and why data contracts and feature consistency matter as much as model architecture.

\item Learn when an AI system should be simplified, constrained, or rejected because the full operational setting does not support reliable deployment.

\begin{whychaptermatters}
This chapter widens the unit of analysis from model to service. After reading it, students should connect objectives, contracts, thresholds, review paths, monitoring, and rollback before calling a system ready. The system readiness brief extends the reliability review card into the book's final launch-or-no-launch artifact.
\end{whychaptermatters}

The previous chapter argued that reliability lives in the full pipeline, not only in the model. Deployment makes that point concrete. A deployed AI system includes data collection, preprocessing, model inference, thresholds, interfaces, logging, monitoring, fallback behavior, human review, and organizational ownership. A model can be mathematically elegant and still fail as a service because one of those surrounding pieces is missing or poorly designed. This systems view matches long-running warnings about hidden technical debt and the software-engineering realities of ML production \citep{sculley2015,amershi2019}.

The question is therefore broader than whether a model learned a useful mapping from inputs to outputs. The real question is whether the whole decision pipeline improves the task under the actual constraints of deployment. That is the habit students need if they want to build trustworthy systems rather than impressive demos.

\begin{problemsolvinglens}
This chapter asks how a prediction becomes an action inside a real organization. The unit of success is no longer the model checkpoint, but the full system that surrounds it.
\end{problemsolvinglens}

\begin{thinkingtool}
\textbf{Pipeline realism.} Ask what would break when a notebook result becomes a service: contracts, preprocessing, thresholds, review queues, monitoring, rollback, ownership, and organizational support.
\end{thinkingtool}

\begin{thinkingtool}
\textbf{Common failure mode $\rightarrow$ recovery move.} Failure mode: trying to launch a strong offline model before the system has an input contract, escalation path, monitoring triggers, and a named owner. Recovery move: write the service blueprint first, including validation, thresholds, fallback behavior, review-capacity limits, alert conditions, and rollback authority, then judge the model only inside that operational design.
\end{thinkingtool}

Keep the same running case in view from the earlier language, experiments, and reliability chapters: a grounded course-support assistant that routes student messages, retrieves institutional policy, drafts bounded answers for routine cases, and escalates uncertain or high-risk cases to humans. The goal now is to judge whether that system is ready to launch as an actual service rather than as an impressive demo.

That language workflow remains the main capstone because contracts, retrieval, escalation, and review paths are easy to see in it. A second contrast case, the dashboard-camera pedestrian detector from Chapter 8, shows that the same systems questions survive outside educational NLP: what objective matters, what inputs are trusted, when automation is allowed, how failure is detected, and who can intervene.

\begin{example}[A First End-to-End Service Sketch]
For the course-support assistant, the system story can already be written in one pass. Validate the incoming message and context. Retrieve only approved current policy sources. Score whether the case is safe enough to answer automatically. If retrieval is weak, the topic is high risk, or that safety score is too low, escalate to a human instead of answering. Log the decision, the evidence used, and the outcome. Monitor overrides, backlog, and citation failures, and give one named owner the authority to tighten the policy or roll back the service.
\end{example}

\begin{example}[Failure Cascade: Good Model, Bad Service]
Suppose the retrieval contract quietly accepts a stale policy page. The model still produces a confident score, so the threshold layer allows direct answers on routine-looking messages. A small fraction of those answers are now wrong, which triggers more user corrections and human overrides. The override spike overloads the review queue, delays genuinely high-risk cases, and finally forces rollback. Nothing in that cascade required a bad neural model in isolation; it required one broken systems boundary to interact with confidence, policy, and queue capacity.
\end{example}

The system is not the model alone. It is the model plus the input contract, policy layer, review path, monitoring, rollback, and owner. A strong offline score can still sit inside a bad service.

\section{Systems Thinking Changes the Question}

When students first learn machine learning, it is natural to think of the model as the product. In practice, the model is only one component inside a larger service. The service has users, deadlines, costs, review paths, logging requirements, failure cases, and operating assumptions. A classifier with strong test accuracy might still be unusable if it is too slow, if it produces too many low-confidence cases for human review capacity, or if it depends on features that are not available at prediction time.

\subsection{From Model Objective to Service Objective}

A model objective is usually narrow: minimize a training loss, maximize validation accuracy, reduce error on a benchmark. A service objective is broader: improve a workflow while respecting operational limits. The two are related, but they are not identical.

Consider the course-support assistant. The institution does not really want confidence scores or generated answers for their own sake. It wants a process that gives students fast, reliable help on routine questions while preserving human advising capacity for ambiguous, sensitive, or high-risk cases. If the assistant escalates too many messages, advisors cannot keep up. If it escalates too few, the system may answer questions that require human judgment or authority. If its behavior shifts every time a policy page is updated, staff will stop trusting it. Good system design begins when the modeling problem is tied back to that service objective.

\subsection{Service-Level Objectives Make the Objective Operational}

One useful way to make the service objective concrete is to write service-level objectives rather than stopping at general hopes such as ``be accurate'' or ``be helpful.'' A service-level objective states what the workflow must achieve under real operating conditions.

\begin{itemize}[leftmargin=*]
\item maximum response latency for routine questions,
\item minimum escalation recall on high-risk messages,
\item maximum average human-review backlog,
\item minimum grounded-citation coverage on automated answers,
\item maximum severe-error rate before rollback is required.
\end{itemize}

These are not replacements for model metrics. They are the bridge between model behavior and operational value. A model can raise offline answer accuracy slightly while violating the system's response-time or review-capacity objectives. In that case the model improved, but the service did not.

\begin{practicalnote}
If the team cannot name the service-level objectives, it is still too early to say whether a threshold, model family, or deployment plan is good. The model objective becomes meaningful only when it is tied to a workflow target the organization actually cares about.
\end{practicalnote}

\begin{decisionbox}
Before asking ``which model performs best?'', ask ``what decision or workflow is being improved, what capacity exists to act on the output, and what kinds of mistakes are acceptable?'' The model can only be judged inside that broader operational context.
\end{decisionbox}

\begin{practicalnote}
A compact deployment workflow is: state the service objective, write the input contract, choose the operating policy and fallback path, test latency and capacity limits, define monitoring and rollback triggers, and only then decide whether launch is justified. A readiness brief simply writes that deployment argument in one place.
\end{practicalnote}

\subsection{Success Includes More Than Accuracy}

Several non-accuracy constraints routinely matter:
\begin{itemize}[leftmargin=*]
\item \textbf{latency}: how quickly must a prediction be returned?
\item \textbf{throughput}: how many predictions must the system handle?
\item \textbf{review capacity}: how many uncertain cases can humans inspect?
\item \textbf{cost}: what computation, storage, or annotation effort is affordable?
\item \textbf{stability}: do outputs fluctuate so much that users lose trust?
\item \textbf{auditability}: can later investigators explain what happened?
\end{itemize}

These constraints are not secondary implementation details. They change which model class, threshold, interface, or deployment pattern is appropriate. The first deployment question is therefore simple: what must the service achieve before any model can be called good enough?

\section{Requirements Before Training}

Well-designed systems begin with requirements. A requirement is a statement about what the system must accomplish or avoid under real operating conditions. Some requirements are quantitative, such as a maximum response time or minimum recall on a high-risk class. Others are structural, such as requiring human confirmation for severe decisions.

\begin{definition}[Operational Requirement]
An operational requirement is a concrete condition that the deployed system must satisfy, such as a latency bound, a review-capacity limit, a minimum recall target, or a mandatory fallback path.
\end{definition}

\subsection{Worked Example: Queue Capacity Changes the Threshold}

Suppose the advising team behind the course-support assistant can manually inspect at most $30$ escalated messages per day. The system assigns each incoming message an \emph{automation-safety score}, where larger values mean stronger evidence that the system can answer safely without human review. On a $200$-message day, a stricter threshold for automatic answering sends $78$ messages to humans. A more permissive threshold sends $26$.

If the team deploys the stricter policy because it appears safer in isolation, the queue may overload the staff and slow all responses, including the truly urgent ones. The seemingly more cautious threshold can therefore produce the weaker service. The stricter threshold may reduce some automation errors on borderline cases, but it also produces a queue the organization cannot actually process.

\begin{mathlens}
Let $q_t$ be the review backlog after day $t$, let $r$ be daily human capacity, and let $a(\tau)$ be the number of cases escalated by automation threshold $\tau$. Then
\[
q_{t+1}=\max\{0,\;q_t+a(\tau)-r\}.
\]
An overly strict threshold can destabilize the queue if it makes $a(\tau)>r$, because the backlog then grows instead of clearing.
\end{mathlens}

This is not a trick example. Many deployments face exactly this issue. The useful prediction is not the one that looks best in isolation. It is the one that leads to the best action process under real constraints.

\begin{example}[Contrast Case: Capacity Looks Different in Vision]
For a pedestrian-alert system, the same requirement may appear as braking latency or the maximum number of false alerts per hour before drivers stop trusting the system. The resource is not an advising queue, but the systems lesson is identical: operating policy must fit the real capacity of the downstream process.
\end{example}

\begin{prereqbridge}
Precision and recall describe statistical behavior. Capacity tells us whether the downstream organization can act on the predictions. A system designer must connect the two. A threshold that increases recall may still be unacceptable if it creates more positive predictions than the workflow can absorb.
\end{prereqbridge}

\subsection{Requirements Should Be Written Down Early}

Teams often delay requirement writing because it feels less exciting than training models. That is backwards. If severe errors, response-time limits, or review paths are not defined early, then later evaluation becomes ambiguous. A model cannot be ``good enough'' until the system has stated what ``enough'' means. Once those requirements are explicit, the next systems question is whether the served inputs actually satisfy the assumptions under which the model was evaluated.

\section{Data Contracts and Training-Serving Consistency}

Many system failures occur even when the model itself is reasonable. The problem is that the data seen at serving time does not match what the training pipeline assumed. Features may be computed differently, missing values may be encoded differently, text may be tokenized inconsistently, or a timestamp field may be shifted by one unit. These failures are often called \emph{training-serving skew} \citep{sculley2015}.

\begin{definition}[Training-Serving Skew]
Training-serving skew is a mismatch between the data representation or feature-processing pipeline used during model development and the one used during deployment.
\end{definition}

\subsection{Why Small Pipeline Mismatches Matter}

Modern models are sensitive to representation. A linear classifier expects features on the scale it was trained on. A tree expects the same fields and semantics it saw during fitting. A neural model expects the same tokenization, image normalization, or waveform preprocessing. If the serving pipeline silently changes one of those steps, the deployed system is effectively running a different model from the one that was evaluated offline.

\begin{mathlens}
If training standardizes a feature as $z=(x-\mu_{\text{tr}})/\sigma_{\text{tr}}$ but serving uses $\tilde z=(x-\mu_{\text{sv}})/\sigma_{\text{sv}}$, then
\[
\tilde z=\frac{\sigma_{\text{tr}}}{\sigma_{\text{sv}}}z+\frac{\mu_{\text{tr}}-\mu_{\text{sv}}}{\sigma_{\text{sv}}}.
\]
For a linear score $s=wz+b$, the served score becomes $\tilde s=w\tilde z+b$, so a changed mean or scale shifts the effective decision boundary even though the weights are unchanged.
\end{mathlens}

\subsection{Worked Example: A Retrieval Contract Bug}

Suppose the course-support assistant was evaluated with a retrieval pipeline that indexed the current semester's official policy pages, chunked them by section, and attached course and term metadata to each chunk. At serving time, a new ingestion job accidentally mixes last semester's pages with the current ones and drops part of the course metadata. The same student question about an extension now retrieves an outdated late-policy passage because the serving index no longer matches the evaluated one.

The consequences can be serious. A message that should have triggered a current-semester escalation path may now receive a confident but stale answer. Nothing is wrong with the language model weights. The system failed because the evidence contract was broken.

\begin{failurebox}
Training-serving skew is dangerous because it often leaves the model file unchanged. Teams may say ``we deployed the validated model'' even though the effective input representation at serving time is different from the one that was validated.
\end{failurebox}

\begin{example}[Real-World Callout: Hidden Technical Debt]
Sculley et al.\ argue that many machine-learning failures come not from the model alone, but from undeclared data dependencies, brittle feature pipelines, feedback loops, and glue code that quietly changes the effective system \citep{sculley2015}. Amershi et al.\ describe similar production realities in large software organizations, where data verification, configuration management, and monitoring often dominate the real engineering burden \citep{amershi2019}. The practical lesson is that a clean notebook result can still hide a brittle service.
\end{example}

\subsection{Data Contracts Reduce Ambiguity}

A practical response is to define data contracts. These specify what features exist, how they are computed, what units they use, which values are allowed to be missing, how categorical values are encoded, and what validation checks must pass before inference occurs. In mature systems, model deployment is blocked when these contracts fail. Requirements state what the service must achieve; contracts state what must be true before the model is allowed to act.

\begin{practicalnote}
Many deployment bugs are ordinary software bugs: wrong field order, missing normalization, stale joins, silently changed units, or inconsistent tokenization. Systems engineering matters because these bugs can dominate model quality in production.
\end{practicalnote}

\section{Prediction Is Not the Decision}

One of the most important system lessons is that a model output is not the final decision. Most models return a score, probability, rank, or generated candidate. The deployed system must still decide what action to take with that output. After the service objective, requirements, and input contract are fixed, this is the next design boundary: how does a score become an action in the real workflow?

\subsection{Thresholds Are Policies}

Choosing a threshold is not a purely mathematical cleanup step after training. It is a policy that trades off false positives, false negatives, workload, and risk. In a low-stakes recommendation setting, one threshold may be acceptable. In a high-stakes screening setting, a different threshold or a mandatory human review path may be required.

\subsection{Queueing, Review Capacity, and Human Bottlenecks}

Once abstention or escalation is available, the threshold also becomes a queueing decision. The same safety score that controls automation coverage also controls review load. If the threshold for automatic answering is raised, more borderline cases move into the human queue. That may reduce some automated mistakes, but it also increases delay, reviewer fatigue, and the chance that urgent cases wait behind routine ones.

This is why queueing belongs in the policy discussion rather than in an operations appendix. The right threshold is constrained by both mistake costs and review capacity. A policy that looks safer on isolated examples may still produce a weaker service once backlog and delay are counted as part of the outcome.

\begin{decisionbox}
When human review is part of the system, do not choose the threshold from precision and recall alone. Estimate how many cases the policy will defer per hour or per day, compare that with real review capacity, and ask what backlog the organization can tolerate before service quality degrades.
\end{decisionbox}

\subsection{Ranking Allocates Exposure}

In search and recommendation systems, the deployed action is often not ``approve or reject'' but ``which items will users actually see, and in what order?'' A ranked list allocates attention. Items shown higher receive more clicks, views, purchases, or follow-up actions partly because they were exposed earlier.

Even the running support-assistant case contains a smaller version of this issue: ranking retrieved passages changes which evidence the generator or the human reviewer sees first, so exposure logic is not only a recommender-system problem.

This makes ranking systems different from ordinary static prediction tasks. Joachims et al.\ show that clickthrough data is informative but biased by position and presentation \citep{joachims2005}. Chapter 1 made the same point from the data side for recommendation logs: observed interactions are filtered by earlier exposure decisions rather than sampled neutrally from all possible user-item pairs \citep{autodebias2021}. A deployed ranking therefore changes the future data that the next model version will see.

The practical consequence is that teams should not monitor only click-through or engagement rate. They should also monitor exposure concentration, subgroup or catalog coverage, repeated overexposure of already popular items, and whether the ranking is narrowing what users get to see. In systems language, a recommender or retrieval ranker is not merely predicting preference. It is shaping the information environment.

\subsection{Selective Prediction and Abstention}

Sometimes the right system does not force the model to answer every case. Instead, it allows abstention: confident cases are automated, while ambiguous cases are deferred to humans or alternative procedures \citep{geifman2017}. In the running support-assistant case, the real decision is often among answer, refuse, and escalate rather than simply positive versus negative.

The word \emph{confidence} needs discipline here. It should not mean a vague feeling that the model ``sounds certain.'' It should mean an estimate tied to the action policy. For the support assistant, a useful quantity is an \emph{automation-safety score}: the estimated probability that an automatic answer will stay in scope, remain grounded in approved evidence, and avoid an escalation-worthy mistake.

This is the same safety idea used in the earlier service sketches. Larger values mean stronger evidence that automatic answering is acceptable. Lowering the threshold accepts more cases automatically; raising it sends more cases to review.

\begin{mathlens}
Let $a(x)\in\{0,1\}$ indicate whether the system answers on input $x$, and let $\ell(f(x),y)$ be the loss on answered cases. Then coverage is $\phi=\mathbb{E}[a(x)]$ and selective risk is
\[
R_\phi(f)=\frac{\mathbb{E}[a(x)\,\ell(f(x),y)]}{\mathbb{E}[a(x)]}.
\]
On this automation-safety score, lowering the threshold usually raises coverage, but it can also raise selective risk if the newly accepted cases are too uncertain. This is why thresholding works best when confidence is reasonably calibrated.
\end{mathlens}

\begin{definition}[Selective Prediction]
Selective prediction is a system design in which the model predicts only on cases where it is sufficiently confident and abstains or defers on the rest.
\end{definition}

\subsection{Worked Example: Confidence, Coverage, and Review}

Suppose the course-support assistant receives $100$ messages:
\begin{itemize}[leftmargin=*]
\item $60$ are routine questions with strong retrieval agreement and an estimated automation-safety score above $0.90$
\item $25$ have moderate scores between $0.60$ and $0.90$ because the wording is unusual or the retrieved evidence is incomplete
\item $15$ contain conflicting signals, policy ambiguity, or high-risk language and are escalated regardless of score
\end{itemize}

If the system answers only the most confident $60$ cases automatically and defers the remaining $40$, then automation coverage is $0.60$. If the automated subset is highly accurate and well grounded, and the human review process handles the deferred cases well, this design may outperform full automation on severe-error control even though fewer cases are answered automatically.

Notice what the score means in this example. A low automation-safety score does not mean ``negative class'' in the ordinary classification sense. It means ``not safe enough to answer automatically under the current policy.''

The right question is therefore not ``can the model answer every message?'' but ``which messages should the system answer on its own, and which should trigger review or fallback?'' That is a systems question. Once that action policy is written, the next question is how the team will notice when the live system drifts away from the assumptions that made the policy look acceptable.

\begin{advancednote}
Selective prediction works best when confidence scores are reasonably calibrated. If predicted probabilities do not reflect real uncertainty well, then thresholding by confidence may not separate easy and hard cases as cleanly as expected.
\end{advancednote}

\section{Monitoring After Deployment}

Deployment is not the end of evaluation. Once a system is live, the environment changes. User behavior changes, interfaces change, sensors degrade, workload grows, and downstream policies change. Monitoring is the mechanism by which the system notices that reality has drifted away from the assumptions made during development. It is what turns pre-deployment assumptions into live checks rather than launch-time promises.

\subsection{What Should Be Monitored}

Useful monitoring often includes:
\begin{itemize}[leftmargin=*]
\item \textbf{input quality}: missing fields, schema violations, extreme values
\item \textbf{distribution change}: shifts in feature statistics or class balance
\item \textbf{prediction behavior}: score distributions, abstention rates, subgroup gaps
\item \textbf{exposure dynamics}: position bias, concentration of traffic, subgroup or catalog coverage in ranked systems
\item \textbf{operational metrics}: latency, throughput, system errors, review backlog
\item \textbf{outcome metrics}: post-deployment accuracy or task success when labels eventually arrive
\end{itemize}

For the course-support assistant, useful monitors would include policy-retrieval hit rate, citation coverage in answered messages, escalation rate by subgroup, advisor override rate, backlog length, and complaint or re-contact rate after automated replies.

\subsection{Delayed Labels Make Monitoring Harder}

Many real systems do not receive immediate ground truth. A support assistant may learn that an answer was inadequate only when an advisor corrects it later, when a student asks again, when a complaint arrives, or when a periodic audit catches a policy mistake. This delay means teams must monitor proxy signals as well as eventual outcomes. A silent increase in retrieval misses, escalation backlog, or answer overrides can be an early warning even before formal labels arrive.

\begin{evidencebox}
Evidence for deployment should include a monitoring plan, alert thresholds, and a rollback path. A system without post-deployment visibility is not really under control, even if its offline evaluation was careful. Production rubrics such as the ML test score make monitoring, rollback, and ownership first-class deployment requirements rather than cleanup tasks \citep{breck2016}.
\end{evidencebox}

\subsection{Rollback and Recovery}

Good systems assume that some deployments will go wrong. They therefore include ways to disable the model, revert to an earlier version, or fall back to a safer baseline. In some settings the fallback may be human review. In others it may be a conservative rule-based heuristic. Recovery planning is part of reliability, not a sign that the model is weak. The next section asks how to generate that post-launch evidence carefully instead of exposing the full user population all at once.

\section{Post-Deployment Experimentation}

Launching a new model to everyone at once creates two avoidable problems. First, it maximizes the blast radius if the new system behaves badly. Second, it makes it harder to learn whether the change actually improved the workflow under real usage. A safer pattern is staged release: observe the system without acting, expose a small slice of traffic first, and expand only when the evidence remains favorable.

\subsection{Shadow Mode and Canary Release}

One common first step is \emph{shadow mode}. The new model runs on live inputs, but its outputs are logged rather than used to drive the user-facing decision. This allows the team to compare the new system with the current one under realistic traffic, latency, and data-quality conditions before users are exposed to the behavior. Shadow mode is especially useful for catching missing fields, stale joins, calibration drift, or review-load surprises that were not visible offline.

If shadow results are acceptable, the next step is often a \emph{canary release}: route a small fraction of real traffic to the new system while monitoring severe-error indicators, latency, override rate, and rollback triggers closely. The point is not to celebrate partial launch. The point is to limit damage while testing whether the deployment assumptions survive contact with production.

\subsection{Online Experiments and Guardrails}

For product changes where randomized exposure is acceptable, online controlled experiments provide the strongest evidence about whether a deployment helps real users. Kohavi et al.\ argue that trustworthy online experiments are valuable precisely because intuition and offline metrics can be misleading once a change reaches live traffic \citep{kohavi2012,kohavi2013}. In this chapter's language, an A/B test is not only a product-growth tool. It is a way to test whether a systems change improves the real workflow without silently degrading something else.

That requires guardrails. A system should not be judged only on one headline metric such as click-through rate, average response time, or overall resolution rate. A rollout scorecard should also track severe failures, subgroup disparities, complaint rate, human-review backlog, safety-rule violations, and any metric that would justify immediate rollback. A small gain on a surface metric does not justify launch if a guardrail metric gets worse.

\begin{decisionbox}
Do not ship a new model to everyone simply because offline validation improved. Ask what staged release plan will test the change safely: shadow evaluation, canary traffic, randomized comparison, guardrail thresholds, and rollback conditions.
\end{decisionbox}

\subsection{Not Every System Should Be Randomized}

Some systems are too high-stakes, too asymmetric in harm, or too constrained by policy to support ordinary online experiments. In those settings the right pattern may be shadow evaluation, restricted rollout to low-risk cases, or delayed deployment until stronger evidence is available. The systems lesson is the same either way: deployment should generate evidence deliberately, not by exposing everyone to a poorly understood change.

\section{Human-in-the-Loop Design}

Human-in-the-loop design is often misunderstood. It does not simply mean adding a human somewhere in the pipeline. The question is whether the model and the human are arranged so that the combined system is better than either one alone. In this chapter's workflow, human review is not an afterthought added to rescue a weak model. It is one of the main ways a system turns uncertain predictions into bounded actions.

\subsection{Humans Need the Right Interface}

If the system presents confusing scores, hides uncertainty, or overloads reviewers with too many cases, then human oversight may be ineffective. A reviewer needs the right information at the right time: the model score, the reason for referral, the relevant features or examples, and the allowed actions. In the support-assistant case, that means the student's message, the retrieved policy passages, the draft answer if one exists, and the rule or uncertainty trigger that caused escalation.

\subsection{Humans Also Introduce Failure Modes}

People can become over-reliant on model outputs, ignore contradictory evidence, or become inconsistent under time pressure. This is sometimes called automation bias. A badly designed interface can therefore make the combined system worse than either the human or the model on its own.

\begin{decisionbox}
Adding a human reviewer does not automatically make a system safe. The review step must have clear triggers, enough time, adequate information, and authority to override the model when necessary.
\end{decisionbox}

\section{Documentation, Ownership, and Iteration}

A system that nobody owns will decay. Inputs change, staff turnover occurs, and the assumptions that were obvious during development are forgotten. Documentation helps preserve institutional memory, but documentation alone is not enough. Teams also need clear ownership of retraining, alert handling, audit requests, and rollback decisions. This is how a launch plan becomes a maintained service rather than a one-time demo.

\subsection{What Documentation Should Capture}

Useful documentation records:
\begin{itemize}[leftmargin=*]
\item the task definition and intended use
\item the training data sources and known limitations
\item the feature pipeline and preprocessing assumptions
\item the evaluation protocol and operating threshold
\item the staged release plan, guardrail metrics, and rollback criteria
\item subgroup, robustness, privacy, and safety results
\item the monitoring plan and fallback behavior
\end{itemize}

This is not bureaucracy for its own sake. It is part of making the system understandable, maintainable, and auditable.

\subsection{Iteration Should Be Scientific}

Teams often respond to a production issue by making a fast change and hoping for improvement. A better pattern is scientific iteration: state the failure, hypothesize the cause, change one part of the system deliberately, reevaluate, and monitor whether the change improved the right metric without creating a worse problem elsewhere.

\section{Chapter Artifact: A System Readiness Brief}

Before a launch decision is credible, the team needs one short document that states who owns the system, what it must achieve, what inputs and policies it relies on, and what would block release. That document is the system readiness brief. It plays the same role for deployment that the framing memo, evaluation plan, claim sheet, and reliability review card played earlier: it turns a broad judgment into something concrete enough to inspect. Production checklists such as the ML test score emphasize many of the same questions \citep{breck2016}.

\begin{example}[Real-World Callout: Readiness Is a Launch Gate]
Breck et al.\ propose the ML test score as a production rubric precisely because offline accuracy is not enough to justify launch \citep{breck2016}. Their checklist asks whether the system has testable data assumptions, serving consistency, monitoring, reproducibility, and operational ownership before users are exposed to it. In this chapter's language, the readiness brief is not classroom paperwork. It is a compact no-launch gate for catching missing rollback paths, broken contracts, or absent owners before deployment turns those omissions into user-facing failures.
\end{example}

\section{When Not to Build an AI System}

The final systems lesson is restraint. Not every decision problem should become a machine-learning deployment. A strong readiness process must allow ``do not launch'' as a technical conclusion rather than treating deployment as the default reward for finishing a model.

\subsection{Simple Rules Sometimes Win}

If a stable rule solves the task well, then adding ML may only increase complexity, opacity, and maintenance burden. This is especially true when the input space is narrow and the desired behavior is already well specified.

\subsection{The Organization May Lack the Needed Support}

Even a promising model should not be deployed if the surrounding organization lacks feature quality checks, review capacity, monitoring, or ownership. In that case the limiting factor is not algorithmic performance. It is system readiness.

\subsection{High Stakes Raise the Evidence Bar}

Some settings demand stronger evidence, stronger fallback behavior, or stronger privacy protection than a team can currently provide. In those settings, the correct decision may be to delay deployment, narrow the use case, or reject the system entirely.

\section{Worked System Walkthrough: Course Support}

The grounded course-support pipeline is a good place to see these pieces together because its launch decision depends on contracts, routing, review, and monitoring all at once. What matters is not just whether the model can answer some messages well, but whether the whole service can be launched, monitored, and corrected responsibly.

\subsection{Step 1: Define the Service Objective}

The institution's goal is not simply to maximize answer accuracy. It wants to help students get fast, reliable help on routine questions while reserving human attention for ambiguous, sensitive, or high-risk cases. The service succeeds only if response time improves, advisor workload stays manageable, and severe escalation misses are kept acceptably low.

\subsection{Step 2: Define the Input Contract}

The system receives the student's message text, course or program context when available, the current term, and a validated set of approved policy documents. Each field has a missing-value rule, format requirement, and freshness check. If the current-semester policy pages are stale, a course identifier is absent, or a document source fails validation, the system should not quietly answer anyway.

\subsection{Step 3: Train and Evaluate the Components}

The team evaluates routing, retrieval, and answer quality chronologically rather than randomly, because the real task predicts future student traffic from past semesters. It reports subgroup metrics, calibration or confidence quality, citation quality, and robustness under wording shift, document reorganization, and missing-context scenarios.

\subsection{Step 4: Choose the Action Policy}

The system does not answer every message. It answers only when the request is in scope, the retrieved evidence comes from approved current sources, the confidence is high enough, and no escalation rule fires. Messages with policy ambiguity, weak evidence, unusual phrasing, or severe-risk content are routed to human staff. This policy is chosen because advising capacity is limited and because ambiguity is exactly where human authority matters.

\subsection{Step 5: Design the Review Interface}

When the system escalates a case, the advisor should not receive only a bare risk score. The review interface should show the student's message, the retrieved policy passages, the draft answer if one was prepared, the uncertainty or rule trigger, and the permitted actions. Otherwise the review step becomes a bottleneck or a rubber stamp rather than a real safeguard.

\subsection{Step 6: Monitor and Iterate}

After deployment, the team monitors document freshness, retrieval hit rate, subgroup escalation rates, advisor override frequency, backlog growth, and eventual quality audits. If the percentage of escalated cases rises sharply, that may indicate drift, a source-ingestion bug, or a confidence-calibration failure. The team can then investigate before performance collapses.

\begin{example}[Mini Incident Postmortem]
Suppose a canary rollout of the course-support assistant looks fine on offline answer quality, but within two days the advisor-review backlog doubles and override rates spike on accommodation questions. A useful first postmortem would not ask only ``which model score moved?'' It would ask which system assumption broke: perhaps the confidence threshold was too low for that slice, the retrieval source mix changed, or the escalation policy routed too many ambiguous cases without enough staffing. The immediate response is to narrow the canary or roll back direct answers on the affected slice, then inspect the backlog trend, override distribution, and recent retrieval evidence before widening again.
\end{example}

This walkthrough shows the chapter's central point: the useful system is not just the language model or retriever. It is the model plus contracts, routing policy, review interface, monitoring, and organizational ownership.

\subsection{Contrast Case: A Pedestrian-Alert Service}

The same readiness questions survive in the dashboard-camera pedestrian detector from Chapter 8. The service objective is no longer ``answer routine questions quickly.'' It is ``surface likely pedestrian hazards early enough to support a safe braking or alerting decision without flooding the driver or controller with false alarms.'' The input contract includes camera calibration, frame rate, lens geometry, synchronization, and health checks for glare or dropout. The operating policy might require agreement between detection and short-term tracking before a high-confidence alert is issued, with conservative slowdown behavior when the visual evidence degrades.

Monitoring also looks different without changing the systems logic. Useful signals include night-rain miss rate, alert rate by lighting condition, camera-health failures, latency spikes, and disagreement between the detector and downstream tracker. A no-launch trigger might be unacceptable failure on close-range pedestrians under dusk, glare, or unseen-camera slices even if average daytime metrics remain strong. The details differ from the support assistant, but the readiness brief still asks the same questions about requirements, contracts, fallback behavior, monitoring, and ownership.

\section{Common Mistakes in Systems Work}

\begin{mistakebox}
\item \textbf{Treating deployment as a final packaging step.} Deployment is part of the design problem from the beginning.
\item \textbf{Optimizing the model while ignoring the workflow.} A slightly better benchmark score may be irrelevant if the system cannot be run or reviewed safely.
\item \textbf{Ignoring training-serving skew.} A mismatch between training and production preprocessing can break an otherwise good model.
\item \textbf{Equating confidence with correctness.} Confidence only helps when scores are calibrated and fallback exists.
\item \textbf{Deploying without monitoring and ownership.} A system without alerts, rollback, and owners is not robust enough for long-term use.
\item \textbf{Shipping to everyone at once.} Expansion should earn its way through shadow evaluation or canary traffic.
\end{mistakebox}

\section{Closing Perspective}

By the time a model becomes a service, the earlier course tools should line up. The framing memo says what task is worth doing. The evaluation plan and claim sheet say what evidence would justify the claim. The reliability review card asks where that claim could break. The system readiness brief asks whether the surrounding pipeline is strong enough to let the model act in the real world. Taken together, these are not bureaucratic extras. They are the discipline of building systems that improve the task under real constraints without hiding their limitations.

The system readiness brief closes the artifact chain that began with the framing memo. The book's workflow now reads as one packet: frame the task, choose the judgment rule, protect the evidence, diagnose the model, bound the failure, and decide whether launch is justified.

\begin{chaptersummary}
\item A deployed AI system includes data pipelines, feature contracts, thresholds, interfaces, monitoring, fallback behavior, and ownership in addition to the model itself.
\item Operational requirements such as latency, review capacity, and severe-error control should be defined before training begins.
\item Training-serving skew occurs when the deployed input pipeline differs from the evaluated one and can break a system even if the model file is unchanged.
\item Thresholds and selective prediction are policy choices that trade off automation, workload, and risk.
\item Ranking and recommendation systems shape future data because what they expose affects later clicks, interactions, and measured feedback.
\item Monitoring, alerting, and rollback are part of the technical design of a reliable system.
\item Staged release strategies such as shadow mode, canary rollout, and online experiments help teams learn safely before full launch.
\item Human-in-the-loop design succeeds only when the review step has clear triggers, adequate information, and real authority.
\item Some tasks should be simplified, delayed, or rejected because the full system evidence is not strong enough.
\item A system readiness brief makes the deployment claim explicit before launch.
\end{chaptersummary}
